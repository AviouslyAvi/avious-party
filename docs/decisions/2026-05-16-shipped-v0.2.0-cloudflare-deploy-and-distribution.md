## Decisions made

### 1. Distribution mechanism: GitHub Releases zip, not Web Store

Goal was "5 friends, this week," not public launch. Chrome Web Store needs a $5 dev fee, ~1–3 day review, and privacy disclosure paperwork — wrong shape. Unpacked extensions installed via `Load unpacked` in `chrome://extensions` are the shortest path. Friends download a zip, unzip, point Chrome at the folder.

Rejected alternative: have friends `git clone`. Adds a git literacy requirement and a less-clean update story.

### 2. No real auto-update; in-extension banner instead

Chrome's auto-update only fires for Web Store-listed extensions. Self-hosted CRX with `update_url` is blocked by consumer Chrome. So an in-extension version check is the closest workable thing — it tells the user when to manually re-download.

Implementation: on boot, `main.ts` fetches `https://api.github.com/repos/AviouslyAvi/Watch-Party/releases/latest`, compares `tag_name` to the baked `VERSION` constant, and if newer shows a blue banner in the panel linking to the release page. Cached in localStorage for 6 hours so we don't hammer GitHub's unauthenticated API (60 req/hr/IP limit).

### 3. SQLite-backed Durable Objects, not legacy storage

First deploy of the relay errored: legacy Durable Objects (`new_classes` migration) require Workers Paid plan. SQLite-backed DOs (`new_sqlite_classes`) are available on the free plan and the Room class uses zero persistent storage anyway (in-memory only per `relay/CLAUDE.md`). One-line edit in `wrangler.toml`.

### 4. Open relay for now, gated later

Anyone with a room URL can join. For 5 friends this is fine — room URLs are 8-char random IDs and only shared in private messages. Gating (shared secret, friend allowlist, per-room password) deferred until distribution widens.

### 5. Cloudflare Pages for the landing page, not GitHub Pages

Account consolidation — the relay was already on Cloudflare, so adding Pages keeps everything in one dashboard with one CLI (`wrangler`). Edge network is also broader than GitHub Pages.

### 6. Option C (GitHub Actions) for future auto-deploy, over Option B (Cloudflare dashboard Git integration)

Chose `.github/workflows/deploy.yml` over Cloudflare's dashboard-managed Git integration because:
- Three deploy targets (landing, relay, extension release) — one workflow file handles all three, two separate Cloudflare integrations handle two of them.
- Config-in-repo matches the project's existing CLAUDE.md / CONTEXT.md / HANDOFF culture.
- Repo is public, so GitHub Actions minutes are unlimited.

See `docs/HANDOFF-github-actions-cicd.md` for the implementation plan and the Cloudflare API token setup that Avi needs to do before the workflow can be written.

## Release process (manual, until CI lands)

1. Edit code on a feature branch, merge to `main`.
2. Bump `client/extension/manifest.json` → `version`.
3. `WS_URL=wss://avious-party-relay.avibenabram.workers.dev npm run build:ext`.
4. `cp dist/extension/content.js dist/extension/manifest.json extension-build/`.
5. `cd extension-build && zip -r ../dist/watch-party-extension-v<VERSION>.zip . -x ".DS_Store"`.
6. Commit + push.
7. `gh release create v<VERSION> dist/watch-party-extension-v<VERSION>.zip --title "v<VERSION>" --notes "..."`.

Friends running an earlier version see the update banner the next time they load a page; click → download → reload unpacked.
